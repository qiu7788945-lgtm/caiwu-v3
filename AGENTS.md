# AGENTS.md

## 项目目标
本项目的长期目标是做成一款可商用的软件。
优先保证稳定、可控、可回退，而不是追求炫技或大规模重构。

## 核心原则
1. 尽量保持本地运行
2. 优先离线可用，非必要不引入联网依赖
3. 不修改现有 UI
4. 不破坏、不重构、不替换已经成型并可正常工作的功能
5. 所有改动以“局部修改、小步推进、可验证、可回退”为原则

## 修改规则
- 默认先阅读和分析，不直接动代码
- 未经明确批准，先输出：
  1. 当前逻辑理解
  2. 准备修改的文件
  3. 修改方案
  4. 风险点
- 只有在用户明确确认后，才开始真正修改代码
- 修改时只改与任务直接相关的最少文件
- 禁止为了“代码更优雅”而擅自重构
- 禁止擅自改变交互方式、页面布局、按钮位置、弹窗样式、表单结构
- 禁止擅自删除已有功能
- 禁止擅自新增复杂依赖、云服务、远程接口
- 如确需联网，必须先说明原因和替代方案

## 产品约束
- 本项目未来应尽量支持离线运行
- 如必须联网，只能用于确有必要的能力，并优先保留本地兜底方案
- 优先考虑桌面端、本地数据库、本地文件处理、本地 OCR、本地缓存
- 任何影响商用稳定性的实验性改动都应避免直接落入主流程

## 前端规则
- 不改现有 UI
- 不重做视觉层
- 不替换现有页面结构
- 不因为“统一风格”而批量改组件
- 只允许做与当前任务直接相关的最小逻辑修改

## 后端与功能规则
- 不动已经成型且验证通过的功能，除非用户明确要求
- 修复 bug 时优先局部修复，不连带改动其他模块
- 新功能接入时优先复用现有结构
- 数据结构变更必须先说明影响范围
- 涉及 OCR、字段映射、财务计算、数据一致性时，必须优先保证结果准确和可解释

## 输出要求
每次分析或修改后，都必须明确说明：
1. 读取了哪些文件
2. 改了哪些文件
3. 每个修改的目的
4. 是否影响现有功能
5. 是否影响离线能力
6. 风险点和回退方式

## 禁止事项
- 不要大规模重构
- 不要擅自优化为“全新架构”
- 不要引入不必要的联网依赖
- 不要擅自改 UI
- 不要擅自删除旧逻辑
- 不要在未确认前直接开始写代码

## 推荐工作流程
第一步：先阅读相关文件并总结当前逻辑  
第二步：列出修改计划和涉及文件  
第三步：等待用户确认  
第四步：执行最小改动  
第五步：汇报改动与验证方式
