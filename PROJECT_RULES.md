你后续对这个项目的所有分析和修改，都必须以“稳定、可控、可回退”为最高原则。

# 三、允许做的事情

你可以做这些事情，但必须以不影响现有功能和 UI 为前提：

1. 对现有逻辑进行保守式优化

# 四、当前阶段的正确优先顺序

当前项目的优先顺序必须按下面执行：

第一优先级：
【OCR 识别链路的分析与优化】

第二优先级：
【本地存储方案的梳理与后续改造规划】

第三优先级：
【测试残留、异常硬编码、假数据来源、无用 AI 代码的识别与精准清理】

注意：
当前不是全面大扫除阶段。
不能一上来就清理旧代码、测试数据或 AI 代码。
必须先围绕 OCR 和数据链路分析清楚，再判断哪些东西可以删、哪些不能删。

# 五、关于 OCR 的特殊说明

OCR 是当前阶段最重要的任务。
后续分析和修改时，要优先围绕以下目标展开：

1. 找出 OCR 识别异常值的来源
2. 找出为什么系统会生成明显不真实的识别内容（例如“31”之类）
3. 区分：
   - OCR 本身识别错误
   - 代码提取逻辑错误
   - 默认值/硬编码污染
   - 测试阶段残留逻辑干扰
4. 在不影响现有 UI 和其他功能的前提下，逐步优化 OCR 识别准确性

注意：
OCR 优化必须是渐进式的、保守式的。
不能为了优化 OCR 而顺手改动大量无关逻辑。

# 六、关于本地存储的特殊说明

本地存储是当前项目后续重点改造对象，但不是第一步就直接重做的对象。
你需要在后续分析中持续关注以下问题：

1. 当前正式运行时，数据到底以哪一套存储为主
2. IndexedDB / Dexie 与 SQLite 是否并存
3. OCR 新增的数据最终写入到哪里
4. 编辑、删除、导出、备份分别依赖哪一套存储
5. Electron 打包后，真实用户环境下的数据链路是否一致
6. 未来如果要改本地存储，应该怎样做到不破坏现有功能和 UI

注意：
在没有彻底分析清楚之前，不允许直接重做本地存储方案。

# 七、关于测试残留、硬编码、无用 AI 代码的说明

项目当前可能存在以下问题：

1. 测试阶段残留的数据
2. 写死的公司名称、假值、默认值
3. 某些异常识别结果来源于历史测试逻辑
4. 代码中可能残留 AI 相关接口、依赖或调用
5. 某些旧逻辑可能表面上看像垃圾代码，但实际仍参与当前正式功能

因此，对这类内容的处理规则是：

第一步：先识别
第二步：确认是否参与当前正式链路
第三步：确认删除后不会影响现有功能和 UI
第四步：再做精准清理

注意：
不能把“看起来没用”当成“真的没用”。
